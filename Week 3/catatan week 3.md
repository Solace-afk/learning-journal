# Jurnal Belajar: Minggu 3

**Topik:** Binary & Executable dalam Reverse Engineering

## Apa yang Saya Pelajari
Materi difokuskan pada pemahaman tentang fondasi dasar dari program komputer, yaitu *binary* dan *executable*. Saya mempelajari bahwa *binary* adalah file murni dalam bentuk kode mesin yang terdiri hanya dari angka 0 dan 1, sehingga tidak dapat dibaca secara langsung oleh manusia. Sementara itu, *executable* adalah file *binary* berstatus *runnable* yang siap dijalankan oleh sistem karena telah diorganisasikan dengan struktur khusus yang mengandung instruksi program, data, dan metadata pendukung. Secara spesifik struktur anatomi dari *executable* mencakup *Header* sebagai "daftar isi" informasi file, *Code Section* yang berisi instruksi perakitan inti, *Data Section* untuk menyimpan variabel global dengan nilai awal seperti *string hardcoded*, *BSS Section* untuk alokasi variabel tanpa nilai awal, serta *Stack* dan *Heap* untuk kebutuhan memori saat *runtime*. Selain itu, setiap sistem operasi memiliki format *executable* uniknya masing-masing, seperti format PE (berbasis *section*) untuk Windows, format ELF (berbasis *segment*) untuk Linux, dan format Mach-O untuk sistem operasi macOS.

## Apa yang Saya Pahami
Melalui materi ini, saya memahami alasan logis mengapa *reverse engineering* difokuskan pada proses analisis tingkat *binary*. Pendekatan ini dilakukan mengingat *source code* atau kode sumber asli dari sebuah aplikasi seringkali tidak tersedia atau memang disembunyikan oleh pihak pengembang. Dengan melakukan tahapan *disassembly* untuk membalikkan *binary* menjadi kode *assembly*, analis dapat memetakan kontrol alur program untuk tujuan keamanan siber, seperti mengetahui mekanisme pengecekan *password*, mendeteksi keberadaan *backdoor*, atau membedah logika *malware* dan mencari kunci dekripsi dari sebuah *ransomware*. Proses ini dijuluki sebagai pekerjaan yang sangat menantang; ketika kode program telah dikompilasi menjadi *binary*, kita akan kehilangan seluruh nama variabel yang deskriptif beserta komentar bantuan dari pembuat program. Terlebih lagi, proses pembacaan ini sering kali dipersulit oleh adanya optimasi *compiler* dan implementasi teknik *obfuscation* atau pengaburan kode yang membuat susunan instruksi tampak acak dan jauh berbeda dari kode sumber aslinya.

## Apa yang Masih Membingungkan
Meskipun secara teoritis struktur dari sebuah *executable* dapat sedikit saya pahami, saya masih bingung mengenai operasional nyata dari area *BSS Section*. Karena disebutkan bahwa bagian BSS (berisi variabel tanpa inisialisasi nilai awal) tidak disimpan secara eksplisit sebagai data fisik di dalam ruang *disk* melainkan hanya dialokasikan secara dinamis saat *runtime* atau program berjalan, saya belum bisa membayangkan dengan jelas bagaimana cara melacak lokasi memori bayangan tersebut saat kita kelak menggunakan bantuan *debugger*. Selain itu, penjabaran mengenai perbedaan konseptual arsitektur pemuatan antara format PE Windows dan ELF Linux masih terasa membingungkan bagi saya.

## Kesimpulan Pribadi
Kesimpulan yang saya dapatkan kali ini adalah bahwa kemampuan membedah dan memahami struktur anatomi dari file *executable* merupakan pintu gerbang utama sebelum memulai investigasi *reverse engineering*. Mampu membaca informasi dari area *Header* dan mengisolasi instruksi penting di area *Code Section* adalah langkah kritis dalam mengurai logika program. Terlepas dari berbagai tantangan teknis seperti pengaburan kode, sesi pembelajaran ini juga kembali menggarisbawahi kompas etika yang wajib dijaga oleh setiap periset; seluruh aktivitas bedah *binary* ini harus dilakukan dalam koridor legalitas yang jelas, ditujukan murni untuk kepentingan edukasi atau penelitian keamanan, dan wajib menjauhi segala bentuk aktivitas yang melanggar hak cipta serta ketentuan lisensi perangkat lunak.
